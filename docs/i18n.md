# i18n 规范（多语言开发与验收清单）

> 本规范定义 sgai 项目支持多语言（当前 zh + en，未来可扩展 ja / ko / ...）的方式。新增页面、新增数据、改动共享组件，全部按本规范执行；提交前必须通过 `npm run check:i18n` 验证。
>
> Last updated: 2026-05-03

## 1. 站点基本架构

- 默认语言 `zh`，路径不带前缀（`/policies/`、`/debates/budget-2914/` 等）。
- 其他语言一律加路径前缀（`/en/policies/`、`/ja/policies/` ...）。EN 已上线，JA/KO 等待加进 `LOCALES` 即可。
- 每个 `src/pages/<route>/<file>.astro` 都有 `src/pages/<lang>/<route>/<file>.astro` 镜像页。Markdown 博客同时使用 `src/data/post/<slug>.md`（zh）与 `src/data/post/<lang>/<slug>.md`（其他语言），通过 frontmatter `lang: 'en'` 等区分。
- 所有 i18n 工具函数集中在 `src/i18n/index.ts`：
  - `LOCALES: Lang[]` / `DEFAULT_LOCALE: Lang` — 当前激活的语言列表与默认语言
  - `getLangFromPath(pathname)` → 当前语言（不识别的前缀回落 default）
  - `localePrefix(lang)` → `''`（默认语言）或 `'/<lang>'`
  - `localizedHref(path, lang)` → 给绝对路径加 lang 前缀
  - `unprefixed(path)` → 去掉任意 lang 前缀，恢复 canonical 默认语言路径
  - `pickLocalized(record, baseKey, lang)` → **新形态（推荐）**，N 语言通用
  - `pickLocalized(record, zhKey, enKey, lang)` → 旧形态（保留兼容，仅限 zh+en）
  - `channelLabel(channel, lang)` → SocialChannel 的显示文本
  - `t(lang, key)` → 字典查询（zh 字典是真值，其他语言缺失时回落到 zh）

要新增一门语言（以 `ja` 为例）：

1. `src/i18n/index.ts` 把 `'ja'` 加进 `Lang` 联合类型 + `LOCALES` 数组
2. 在 `FALLBACK_CHAINS` 加 `ja: ['ja', 'en', 'zh']`（或想要的 fallback 顺序）
3. 加导出 `export const ja: Partial<Record<keyof typeof zh, string>> = { ... }` 字典
4. 在数据文件给用户可见的字段加 `*Ja` 兄弟字段（如 `titleJa`、`descriptionJa`）
5. 镜像 `src/pages/<route>` 到 `src/pages/ja/<route>`，模式与 `src/pages/en/` 一致
6. 把 `Metadata.astro` 的 `og:locale` 映射加上 `ja: 'ja_JP'`
7. `npm run build && npm run check:i18n -- --lang ja` 跑验证（需要为 `ja` 在 `LANG_CONFIG` 加配置）

## 2. 数据契约（双字段约定）

> 一句话：数据文件以中文为主语言，其他语言以可选 `*<Cap-Lang>` 兄弟字段并列存储；展示由页面 + `pickLocalized` 决定。

### 2.1 命名规则

- 默认语言（zh）字段用裸名：`title`、`description`、`summary`、`role`...
- 其他语言加 capitalized 后缀：`titleEn`、`titleJa`、`titleKo`、`descriptionEn`...
- 数组字段同样加后缀：`keyDebates: string[]` ↔ `keyDebatesEn?: string[]`。
- TypeScript 接口里非默认语言字段都标 `?` 可选；类型注释加一句"English/Japanese/... sibling of X"。

```typescript
export interface KeyInsight {
  title: string;
  titleEn?: string;
  description: string;
  descriptionEn?: string;
  evidence: string;
  evidenceEn?: string;
  significance: 'high' | 'medium' | 'low';
}
```

### 2.2 例外：标题已是英文的字段

某些数据 `title` 字段本身存的是英文（例如国会辩论 Hansard 原始标题）。此时 `zhTitle` 是中文兄弟字段。这种"英文为主+中文翻译"的反向布局允许存在，但所有页面渲染时**仍然走 `pickLocalized`**，不要假设方向。

### 2.3 状态枚举值

`status: '已落实' | '进行中' | '仍在讨论'` 这种枚举型字符串，比较用原值（中文），但**展示给用户的字符串要在页面层做翻译映射**。具体可参考 `src/pages/en/debates/index.astro` 的 `statusLabels`。

如果将来要彻底干净，把枚举值换成 `'implemented' | 'in-progress' | 'under-discussion'`，再用页面层翻译。当前为兼容历史数据保持中文枚举值。

### 2.4 SocialChannel `label` / `labelEn`

- `label` 默认存 zh（如 `"官网"`、`"MDDI 官网"`、`"@joteo_ylm"`）。
- 若 `label` 含 CJK 字符，**必须**同时给 `labelEn`（英文等价物，如 `"Website"`、`"MDDI website"`）。
- 平台名（twitter、linkedin、facebook 等）由 `channelLabel()` 内置兜底，不需要逐条写。
- 例外：纯拉丁文标签（`@joteo_ylm`、`Newsletter`）不需要写 `labelEn`，渲染时直接复用。

### 2.5 生成型长文本（字幕、transcript、抓取正文）

- 生成型长文本也必须遵守同一套兄弟字段规则，不能因为来源是外部抓取就直接复用到所有语言页面。
- 默认语言字段仍然用裸名，例如视频 transcript 的 `paragraphs` 是中文；英文用 `paragraphsEn`。
- 如果抓到的来源是英文，要先存入 `paragraphsEn`，再翻译成 `paragraphs` 给中文页使用。
- 页面层必须按 lang 选择字段；中文页不得直接把英文 source transcript 当作正文渲染。
- 国会辩论详情页是强约束：英文原文必须直接展开；非英文页面必须额外显示完整译文。中文页展示 `paragraphs`（中文译文）+ `paragraphsEn`（英文原文），英文页只展示 `paragraphsEn`。
- 视频 transcript 有专门检查：`npm run check:video-transcripts`。它已经并入 `npm run check`。
- 国会辩论 transcript 有专门检查：`npm run check:debate-transcripts`。它已经并入 `npm run check`。

## 3. 页面渲染规则

### 3.1 顶层契约

- 每个 `src/pages/<lang>/...` 页面顶部声明 `const lang = '<lang>' as const;` 并把 `lang={lang}` 传给 `Layout`。
- 每个 zh 页面如果使用了任何接受 `lang` prop 的组件，也要传 `lang`（默认从路径推断，但显式传更稳）。
- 渲染数据字段时，**永远不要**直接 `{record.title}` 当默认语言用。统一用：
  - `pickLocalized(record, 'title', lang)` ← **新形态推荐**，自动按 lang 选 `title` / `titleEn` / `titleJa`
  - 简写 `record.titleEn || record.title` 仅在 EN sibling 几乎总存在时使用，且会限制于双语系统
  - 旧形态 `pickLocalized(record, 'title', 'titleEn', lang)` 保留兼容，但新代码不要写

### 3.2 链接

- 跳到内部页面时，永远过一层 `localizedHref(path, lang)`。
- 不要硬编码 `/en/` 前缀；不要硬编码 `/people/...` 这类裸路径。
- 例：`<a href={localizedHref(`/people/${p.id}/`, lang)}>...</a>`

### 3.3 共享组件（zh + EN 都使用）

放在 `src/components/` 下、不分 lang 的组件（Header、Footer、Logo、LangBanner、LanguageToggle、Breadcrumb、AuthorBio、TableOfContents、NextPrevPost、RelatedRail、blog/ListItem、blog/SinglePost），必须**自己**从 URL 推断 lang：

```astro
---
import { getLangFromPath } from '~/i18n';
const lang = getLangFromPath(new URL(Astro.url).pathname);
const isEn = lang === 'en';
---
```

然后所有展示文案、内部链接、字段读取都按 lang 走。

### 3.4 镜像页应是 thin wrapper（v0.6.0 起强制）

> 一句话：任何在多语言下结构相同的页面（首页、about、timeline、人物详情等），**结构必须只存在一份**。每个 lang 的 page 文件只负责 metadata + JSON-LD，body 渲染统一委托给共享组件。

**为什么**：v0.5.0 之前我们把"数据层去重 + 页面层独立"作为分工。结果首页设计迭代两次，EN 都没跟上——`zh` 写的是裸中文字符串、`en` 写的是 `t(lang, ...)`，两份文件长得就完全不像，diff 看不出结构漂移；`i18n-check.mjs` 只查"EN 有没有中文残留"，不查"两边结构是否一致"。漂移是必然，不是意外。

**怎么做**：把页面 body 抽到 `src/components/<area>/<PageName>.astro`，接收 `lang: Lang` prop。两个 page 文件都引入同一个组件，各自只填自己的 metadata。

参考实现 — 首页：

- 共享组件：[`src/components/home/HomePage.astro`](../src/components/home/HomePage.astro)
- ZH 包装：[`src/pages/index.astro`](../src/pages/index.astro)（~40 行）
- EN 包装：[`src/pages/en/index.astro`](../src/pages/en/index.astro)（~40 行）

包装文件模板：

```astro
---
import Layout from '~/layouts/PageLayout.astro';
import JsonLd from '~/components/common/JsonLd.astro';
import MyPage from '~/components/<area>/MyPage.astro';
import { t } from '~/i18n';

const lang = 'zh' as const; // or 'en' as const

const metadata = { title: t(lang, 'siteName'), description: t(lang, 'siteDescription') };
const websiteSchema = { '@context': 'https://schema.org', '@type': 'WebSite', /* ... */ };
---

<Layout metadata={metadata} lang={lang}>
  <JsonLd schema={websiteSchema} />
  <MyPage lang={lang} />
</Layout>
```

共享组件内部规则：

- 所有用户可见文案走 `t(lang, key)` 字典；不要嵌任何裸字符串
- 数据字段走 `pickLocalized(record, baseKey, lang)`
- 内部链接走 `localizedHref(path, lang)`
- 不要在共享组件里 `if (lang === 'zh') ... else ...` 分支结构差异——结构必须一致，差异只在文案

**适用范围**：

- ✅ 首页、about、timeline、tracker overview 这种镜像页
- ✅ 镜像但只是顺序/排版略有不同的页面——把差异收进字典或参数，仍然走共享组件
- ❌ Markdown 博客文章（`src/data/post/<slug>.md` 与 `src/data/post/en/<slug>.md`）——内容本就是各语言独立写的，沿用现有双文件方案
- ❌ EN 独占或 ZH 独占的页面——没有镜像，自然没漂移问题

**新增镜像页的步骤**：

1. 写共享组件 `src/components/<area>/<PageName>.astro`，body + 顶部 `interface Props { lang: Lang; }`
2. 在 zh 字典里把所有用户可见文案登记成 key，EN 字典补对应翻译
3. 在 `src/pages/<route>` 与 `src/pages/en/<route>` 各放一个 thin wrapper
4. 跑 `npm run build && npm run check:i18n` 确认 EN 端无中文残留

**改动镜像页时**：直接改共享组件。**永远不要**只改其中一个 page 文件。如果发现自己在 page 文件里写了任何 body 结构（不只是 metadata），停下来，把它抽到共享组件去。

### 3.5 zh 副字段（zhTitle, zhName, zhEvent, etc.）

- 在 zh 页面：英文条目可保留 zh 翻译做副显示，符合中文阅读体验。
- 在 EN 页面（以及未来的 ja/ko/...）：**完全隐藏** zh 副字段。EN 页面要保持纯英文，不展示中文注音/副标题。一旦在非默认语言页面渲染了 zh 副字段，i18n-check 会报错。
- 这是 v0.5.0 起的新约定（之前曾允许短姓名做注音，已撤销）。

## 4. SEO 与 Metadata

- `<html lang>`：zh 页面输出 `lang="zh-CN"`；EN 页面输出 `lang="en"`。已由 `src/layouts/Layout.astro` 统一处理。
- `og:locale`：zh `zh_CN`、EN `en_US`。由 `src/components/common/Metadata.astro` 根据 lang 输出。
- `og:site_name`：zh `新加坡 AI 观察`、EN `Singapore AI Observatory`。来源是 `src/i18n/index.ts` 字典。
- `<title>` 模板：zh `%s — 新加坡 AI 观察`、EN `%s — Singapore AI Observatory`。
- `description` 兜底：每个页面应自带 EN description；缺失时回落 `t('en', 'siteDescription')`。
- `<link rel="alternate" hreflang>`：zh 页面输出 `hreflang="en"` 指向 `/en/...` 镜像，EN 页面输出 `hreflang="zh-CN"` 指向 zh 镜像。（已在 `CommonMeta.astro` 配置；新增页面如果走 PageLayout 自动继承，无需重复配置。）

## 5. 验证

### 5.1 自动检查（每次构建）

```bash
npm run build
npm run check:i18n               # 默认扫 dist/en
npm run check:i18n -- --lang en  # 显式
# 未来支持其他语言后：
# npm run check:i18n -- --lang ja --root dist
```

脚本逻辑：
1. 走 `dist/<lang>/**/*.html`
2. 剥离 `<script>` `<style>` `<head>` 与所有 HTML 属性
3. 按 `LANG_CONFIG[<lang>].foreignRegex` 搜索"不该出现的脚本"（EN 默认是 CJK Unified Ideographs）
4. 对照 `LANG_CONFIG[<lang>].allowPatterns` 白名单
5. 命中即报错并定位到具体页面

EN 当前白名单（每条都有理由）：
- `中文版可用` / `阅读中文版` / `中`：LangBanner 与 LanguageToggle 邀请用户切换到 zh 的副本。

如果你为某条特殊残留申请例外，**必须**在 `LANG_CONFIG[<lang>].allowPatterns` 里加注释说明原因。

### 5.2 反向检查（zh 页面是否被英文污染）

目前没有自动化的"zh 页面英文残留"检查，因为 zh 页面允许大量英文（人名、机构缩写、产品名）。如果未来需要，可以加一个 `zh-check.mjs` 反向版，但允许列表会很长，价值有限。手工 review 即可。

### 5.3 PR / 提交前必跑

```bash
npm run check          # astro check + eslint + prettier + verify-graph
npm run build          # 必须先构建才能跑 i18n-check
node scripts/i18n-check.mjs   # 中文残留扫描
```

## 6. 新增内容时的清单

### 6.1 新增数据条目（debate / policy / lever / video / 等）

- [ ] zh 字段填全
- [ ] 每个用户可见的 zh 字段都加了 `*En` 兄弟字段（标题、摘要、要点、状态描述、机构名）
- [ ] 数组字段同样加了 `*En` 数组（即使内容相同也要复制一份，防止页面在 EN 下显示 zh）
- [ ] 新增 / 更新国会辩论后运行 `npm run fetch:debate-transcripts && npm run translate:debate-transcripts && npm run check:debate-transcripts`
- [ ] 跑一次 `npm run build && node scripts/i18n-check.mjs` 看新条目是否引入新残留

### 6.2 新增页面

- [ ] `src/pages/<route>/...astro`（zh）和 `src/pages/en/<route>/...astro`（EN）双份
- [ ] EN 页面顶部 `const lang = 'en' as const;` 并传给 Layout
- [ ] 渲染数据走 `pickLocalized` 或 `*En` 兜底
- [ ] 内部链接用 `localizedHref`
- [ ] 跑 `i18n-check` 通过

### 6.3 改动共享组件

- [ ] 用 `getLangFromPath(new URL(Astro.url).pathname)` 推断 lang
- [ ] 显示文案用 `t(lang, key)` 或本地三元
- [ ] 字段取值用 `pickLocalized`
- [ ] 链接用 `localizedHref`
- [ ] zh 与 EN 各跑一遍构建确认两端都对

### 6.4 SEO 调优

- [ ] 修改 `<title>` / description / og 标签时同时改 zh 与 EN 两份字典
- [ ] hreflang 必须同时存在（zh ↔ en 双向）
- [ ] 页面 metadata 不要硬编码"新加坡 AI 观察"或"Singapore AI Observatory"，统一走 `t(lang, 'siteName')`

## 7. 已知限制

- `src/data/post/en/*.md` 的 EN 博客以独立文件存在，与 zh 版手工保持一致；改 zh 文章不会自动同步到 EN，需要双改。
- `LangBanner` 在 EN 页面显示中文邀请文案（`中文版可用`、`阅读中文版`），是有意为之——这是给具有中文浏览偏好的访客看的。i18n-check 已为此加了白名单。
- Hansard 国会辩论的"speakers"字段保持英文姓名（与 Hansard 原文一致）。
